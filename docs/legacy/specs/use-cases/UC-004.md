# Use Case: Activación de Tarjeta de Visitante

## Metadata

| Propiedad     | Valor                                                      |
| ------------- | ---------------------------------------------------------- |
| UC ID         | UC-004                                                     |
| Reglas        | RULE-006, RULE-007                                         |
| Actor primario| Operador de seguridad (usuario humano en terminal)         |
| Alcance       | Sistema — SICAWeb (`ActivarCartoes.ascx`)                  |
| Nivel         | User Goal                                                  |
| Estado        | Draft — extraído de legacy                                 |
| Fuente        | `demo/from_old_src/SICAWeb/SICAWeb/ActivarCartoes.ascx.vb` |

## Descripción

El operador asigna una tarjeta de visitante disponible a un visitante registrado, fijando
la fecha de validez. El sistema determina qué tarjetas están disponibles según si tienen
hora de salida registrada o nunca han sido usadas. Solo las tarjetas asociadas a las familias
de acceso del terminal del operador se muestran.

## Partes interesadas e intereses

| Parte interesada       | Interés                                                              |
| ---------------------- | -------------------------------------------------------------------- |
| Operador de seguridad  | Poder asignar rápidamente una tarjeta disponible a un visitante      |
| Gestión de seguridad   | Las tarjetas en uso no deben reasignarse; las devueltas deben reciclarse |
| Visitante              | Obtener una tarjeta válida con el mínimo de espera                   |

## Precondiciones

1. El operador ha superado UC-003 (sesión con `Session("NomeTerminal")` establecida).
2. Existen familias de acceso asociadas al terminal del operador en `vwFamilias`.
3. Existen tarjetas de visitante registradas en `vwREFERVisitantes`.

## Postcondiciones (garantías de éxito)

1. Una tarjeta previamente disponible queda asignada al visitante.
2. La tarjeta tiene una fecha de validez (`ValidadeCartao`) establecida.
3. La tarjeta deja de aparecer en la lista de disponibles.

## Disparadores

- Carga del control `ActivarCartoes.ascx` desde el panel del operador.

## Flujo principal de éxito

| Paso | Actor    | Sistema                                                                        |
| ---- | -------- | ------------------------------------------------------------------------------ |
| 1    | Operador | Accede al módulo de activación de tarjetas de visitante                        |
| 2    |          | Consulta familias de acceso del terminal: `SELECT IDFamilia FROM vwFamilias WHERE NomeTerminal = nombre` |
| 3    |          | Construye filtro dinámico de familias para `vwREFERVisitantes`                 |
| 4    |          | Para cada tarjeta en `vwREFERVisitantes`, evalúa disponibilidad (RULE-006)     |
| 5    |          | Muestra solo las tarjetas disponibles al operador                              |
| 6    | Operador | Selecciona una tarjeta y asigna al visitante                                   |
| 7    |          | Fija `ValidadeCartao = FormataData(Now)` (RULE-007)                            |
| 8    |          | Guarda la asignación en la base de datos                                       |
| 9    |          | Confirma la activación al operador                                             |

### Detalle de pasos

**Paso 4 — Filtro de disponibilidad (RULE-006)**

Una tarjeta de la vista `vwREFERVisitantes` se considera **disponible** si:

- `HoraSaida.Length > 0` — el visitante anterior ya salió, OR
- `HoraEntrada.Length = 0 AND HoraSaida.Length = 0` — nunca usada.

Una tarjeta se considera **en uso** (no mostrada) si:

- `HoraEntrada.Length > 0 AND HoraSaida.Length = 0`.

**Paso 2–3 — Filtro por terminal (RULE-008 extensión)**

- Solo se muestran tarjetas de las familias asociadas al terminal del operador.
- ⚠️ La consulta de familias concatena el nombre de terminal sin parametrizar
  ([ActivarCartoes.ascx.vb:18](../../../../../demo/from_old_src/SICAWeb/SICAWeb/ActivarCartoes.ascx.vb)).

**Paso 7 — Fecha de validez (RULE-007)**

- `ValidadeCartao = FormataData(Now)` fija la fecha de activación.
- No se observa validación posterior que impida el acceso a una tarjeta caducada.

## Extensiones (flujos alternativos)

### 5a. No hay tarjetas disponibles

| Paso | Actor    | Sistema                                                    |
| ---- | -------- | ---------------------------------------------------------- |
| 5a.1 |          | El filtro de disponibilidad no devuelve ninguna tarjeta    |
| 5a.2 |          | Se muestra lista vacía al operador                         |
| 5a.3 | Operador | Debe esperar a que un visitante devuelva su tarjeta        |

### 4a. Tarjeta con datos inconsistentes (salida sin entrada)

| Paso | Sistema                                                          |
| ---- | ---------------------------------------------------------------- |
| 4a.1 | `HoraSaida.Length > 0` aunque `HoraEntrada.Length = 0`           |
| 4a.2 | Se trataría como disponible (condición de salida cumplida)       |
| 4a.3 | Estado inconsistente no detectado                                |

### 2a. Terminal sin familias asociadas

| Paso | Sistema                                                    |
| ---- | ---------------------------------------------------------- |
| 2a.1 | Consulta de `vwFamilias` devuelve 0 familias               |
| 2a.2 | No se construye filtro → posiblemente no se muestran tarjetas |

### *a. Terminal no autorizado (sesión sin NomeTerminal)

| Paso | Sistema                                                     |
| ---- | ----------------------------------------------------------- |
| *a.1 | `Session("NomeTerminal")` está vacío                        |
| *a.2 | Sin verificación explícita en `ActivarCartoes.ascx`         |
| *a.3 | Comportamiento indefinido (posible acceso sin terminal)     |

## Requisitos especiales

- **Seguridad**: Consulta de familias sin parametrizar (P1). ⚠️
- **Concurrencia**: Sin mecanismo de bloqueo; dos operadores podrían asignar la misma
  tarjeta simultáneamente.
- **Caducidad**: La lógica de validación de `ValidadeCartao` no está implementada en el
  flujo de acceso observado.

## Frecuencia estimada

- Varias veces al día por terminal activo en puestos de recepción y control de accesos.

## Cuestiones abiertas

- ¿`ValidadeCartao` es la fecha de activación o la de caducidad?
- ¿Dónde se valida la caducidad en el paso de entrada física del visitante?
- ¿Existe un proceso de devolución explícito o se gestiona por el hardware de control?
