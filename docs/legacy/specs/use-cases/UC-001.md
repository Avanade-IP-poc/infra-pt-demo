# Use Case: Sincronización de Empleados desde Active Directory

## Metadata

## Descripción

El servicio `SICADataSync` consulta el Active Directory de REFER para obtener todos los
empleados activos de cada organización, valida sus datos de identidad y los replica
(crear o actualizar) en la base de datos SICA. Este proceso mantiene sincronizados los
registros de usuarios entre el directorio corporativo y la plataforma de control de accesos.

## Partes interesadas e intereses

| Parte interesada     | Interés                                                         |
| -------------------- | --------------------------------------------------------------- |
| Operación de Seguridad | El maestro de usuarios SICA refleja el estado actual del AD  |
| RRHH / AD            | Los empleados dados de baja no deben tener acceso activo        |
| Auditoría            | Trazabilidad de altas y modificaciones de usuarios              |

## Precondiciones

1. El servicio dispone de credenciales válidas para leer el AD (`refer.pt`, `DC=refer,DC=pt`).
2. La base de datos SICA (`rfsql01`) es accesible con las credenciales configuradas.
3. Se ha iniciado la ejecución mediante clic en `cmdRun` o parámetro `/START`.

## Postcondiciones (garantías de éxito)

1. Todos los empleados activos y válidos de las 4 organizaciones existen en SICA.
2. Los empleados ya presentes tienen sus datos actualizados.
3. Se envía un informe por correo a `ajsfernandes@refer.pt` con los contadores de la ejecución.

## Disparadores

- Arranque manual del ejecutable con `/START`.
- Pulsación del botón `cmdRun` en la interfaz de la aplicación.
- Tarea programada del sistema operativo (frecuencia no documentada).

## Flujo principal de éxito

| Paso | Actor              | Sistema                                                              |
| ---- | ------------------ | -------------------------------------------------------------------- |
| 1    | Daemon             |                                                                      |
| 2    |                    | Itera sobre las 4 organizaciones: REFER, REFERTelecom, REFERPatrimonio, REFEREngineering |
| 3    |                    | Para cada organización, consulta AD filtrando por `company = org`    |
| 4    |                    | Para cada registro AD recuperado, valida el `employeeID` (RULE-001)  |
| 5    |                    | Si `employeeID.Length ≠ 7` o empieza por `999999/888888/777777` → descarta |
| 6    |                    | Busca en SICA un registro con `codelogique = employeeID` (RULE-002)  |
| 7    |                    | Si no existe: llama `NovoUtilizadorSICA()` → INSERT                  |
| 8    |                    | Si existe: llama `ActualizaUtilizadorSICA()` → UPDATE                |
| 9    |                    | Acumula contadores de altas y modificaciones                         |
| 10   |                    | Finaliza la iteración y envía informe por correo                     |

### Detalle de pasos

**Paso 4 — Validación de Employee ID (RULE-001)**

- Entrada: `drAD("employeeID")` — campo del registro AD.
- Condición aceptada: longitud = 7 AND `Left(6)` NOT IN (`999999`, `888888`, `777777`).
- Motivo del filtro: excluye cuentas genéricas o de servicio.

**Paso 3 — Filtro por organización (RULE-003)**

- Campo usado: `drAD("company")`.
- Valores conocidos: REFER=4, REFERTelecom=1480, REFERPatrimonio=1656, REFEREngineering=1658.
- Cada pasada itera únicamente sobre la organización actual.

**Paso 6–8 — Merge strategy (RULE-002)**

- Clave natural de unión: `codelogique` (= `employeeID` de AD).
- Se usa `dtSICA.Select("codelogique='...'")` para detectar existencia previa.

## Extensiones (flujos alternativos)

### 4a. Employee ID inválido

| Paso | Sistema                                               |
| ---- | ----------------------------------------------------- |
| 4a.1 | `employeeID.Length ≠ 7` o prefijo reservado detectado |
| 4a.2 | Registro descartado; contador de descartados +1        |
| 4a.3 | Continúa con el siguiente registro AD                 |

### 3a. AD no accesible

| Paso | Sistema                                              |
| ---- | ---------------------------------------------------- |
| 3a.1 | `LeUtilizadoresREFER()` lanza excepción de conexión  |
| 3a.2 | Excepción capturada (sin manejo explícito observado) |
| 3a.3 | Caso de uso termina sin sincronización               |

### *a. Base de datos SICA no disponible

| Paso | Sistema                                                      |
| ---- | ------------------------------------------------------------ |
| *a.1 | `InicializeConnSQL` falla                                    |
| *a.2 | Sin manejo explícito de excepciones en `SQLMethods.vb`       |
| *a.3 | Ejecución termina con error no gestionado                    |

## Requisitos especiales

- **Seguridad**: Las credenciales de BD están en texto plano en `app.config:9-11`. ⚠️
- **Rendimiento**: Sin límite de batch; todos los empleados se procesan en memoria.
- **Auditoría**: Solo se registra el contador final en el informe de correo; no hay log por registro.
- **Idempotencia**: Reejecutar produce el mismo estado final (crear o actualizar).

## Frecuencia estimada

- No documentada; se asume ejecución programada (diaria o semanal).
