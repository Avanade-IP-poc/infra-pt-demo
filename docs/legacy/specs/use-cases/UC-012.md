# Use Case: Configuración de Permisos de Acceso por Terminal

## Metadata

| Propiedad        | Valor                                                       |
| ---------------- | ----------------------------------------------------------- |
| UC ID            | UC-012                                                      |
| Reglas           | RULE-008 (control de acceso por usuario autorizado)         |
| Actor primario   | Administrador del sistema (usuario humano autorizado)       |
| Alcance          | Sistema — SICAWeb (`Acessos.ascx`, `ConfigurarAcessos.aspx`, `PagConfigAcessos.aspx`) |
| Nivel            | User Goal                                                   |
| Estado           | Draft — extraído de legacy                                  |
| Fuente           | `demo/from_old_src/SICAWeb/SICAWeb/Acessos.ascx.vb`         |

## Descripción

El administrador configura qué tarjetas, familias y circuitos están permitidos en un terminal
concreto. El sistema sincroniza el catálogo local (BD SICA) con los datos del servicio SMI y
persiste las relaciones tarjeta-terminal. El acceso está restringido a usuarios autorizados.

## Partes interesadas e intereses

| Parte interesada       | Interés                                                         |
| ---------------------- | --------------------------------------------------------------- |
| Administrador          | Definir el perfil de seguridad de cada terminal                 |
| Seguridad              | Garantizar que solo personal autorizado configure permisos      |
| Operadores             | Que cada terminal exponga las tarjetas/circuitos correctos      |

## Precondiciones

1. El usuario está incluido en `AppSettings("AcessoDeConfiguracao")`.
2. El servicio SMI está disponible para obtener tarjetas, familias y circuitos.
3. El operador ha superado [UC-003](UC-003.md) (sesión válida).

## Postcondiciones (garantías de éxito)

1. El catálogo local (`tblCartoes`, `tblFamilias`, `tblCircuitos`) está sincronizado con SMI.
2. Las relaciones tarjeta-terminal (`tblCartoesTerminal`) reflejan la configuración aplicada.

## Disparadores

- Carga del control `Acessos.ascx`.
- Selección de un terminal en el filtro.
- Pulsación del botón "Aplicar".

## Flujo principal de éxito

| Paso | Actor          | Sistema                                                              |
| ---- | -------------- | ------------------------------------------------------------------- |
| 1    | Administrador  | Accede al módulo de configuración de accesos                        |
| 2    |                | Verifica que el usuario está en `AcessoDeConfiguracao` (RULE-008)   |
| 3    |                | Carga terminales (`vwTerminales`), tarjetas, familias y circuitos    |
| 4    |                | Sincroniza el catálogo local con SMI (alta de entidades nuevas)     |
| 5    | Administrador  | Selecciona un terminal                                              |
| 6    |                | Carga las relaciones actuales del terminal                          |
| 7    | Administrador  | Marca/desmarca tarjetas, familias y circuitos permitidos            |
| 8    | Administrador  | Pulsa "Aplicar"                                                    |
| 9    |                | Borra e inserta relaciones en `tblCartoesTerminal`                  |

### Detalle de pasos

**Paso 2 — Control de acceso (RULE-008)**

- `AppSettings("AcessoDeConfiguracao")` define los usuarios autorizados.

**Paso 4 — Sincronización con SMI (⚠️ riesgo de seguridad)**

- ⚠️ **Inyección SQL SEVERA**: `Acessos.ascx.vb:48-49,64-65,78-79` construye sentencias
  `INSERT` concatenando `LogicalCode`/`Label` recibidos de SMI sin escape. Una etiqueta con
  comilla simple (p. ej. `O'Brien`) rompe o inyecta SQL.

**Paso 9 — Persistencia de relaciones**

- `delete`/`insert` sobre `tblCartoesTerminal` filtrando por `SelectedValue` (entero).

## Extensiones (flujos alternativos)

### 2a. Usuario no autorizado

| Paso | Actor          | Sistema                                                |
| ---- | -------------- | ------------------------------------------------------ |
| 2a.1 |                | El usuario no está en `AcessoDeConfiguracao`           |
| 2a.2 |                | Deniega el acceso al módulo                            |

### 8a. Cancelación

| Paso | Actor          | Sistema                                          |
| ---- | -------------- | ------------------------------------------------ |
| 8a.1 | Administrador  | Pulsa "Cancelar"                                |
| 8a.2 |                | Descarta los cambios sin persistir               |

## Requisitos especiales

- **Seguridad SEVERA**: Parametrizar los `INSERT` de sincronización SMI; los datos externos
  deben tratarse como no confiables (riesgo de inyección vía servicio SMI comprometido).
- **Autorización**: Restricción por `AcessoDeConfiguracao`.
- **Consistencia**: La sincronización catálogo↔SMI debe ser idempotente.

## Frecuencia estimada

- Esporádica (alta/cambio de terminales y catálogos).

## Cuestiones abiertas

- ¿La sincronización SMI es incremental o completa en cada acceso?
- ¿Qué ocurre con relaciones huérfanas si una entidad SMI desaparece?
