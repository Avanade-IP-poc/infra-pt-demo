# Use Case: Autorización de Terminal y Sesión de Usuario

## Metadata

| Propiedad     | Valor                                                 |
| ------------- | ----------------------------------------------------- |
| UC ID         | UC-003                                                |
| Reglas        | RULE-008, RULE-009                                    |
| Actor primario| Operador de seguridad (usuario humano en terminal)    |
| Alcance       | Sistema — SICAWeb                                     |
| Nivel         | User Goal                                             |
| Estado        | Draft — extraído de legacy                            |
| Fuente        | `demo/from_old_src/SICAWeb/SICAWeb/Default.aspx.vb`   |

## Descripción

Cuando un operador navega a SICAWeb desde un PC, el sistema verifica dos condiciones de
forma secuencial: que el PC está registrado como terminal autorizado (por hostname o IP) y
que el usuario ha sido identificado mediante Windows Auth. Si ambas condiciones se cumplen,
se establece la sesión y el operador accede al panel de monitorización. Esta es la **única**
frontera de autorización del sistema legacy.

## Partes interesadas e intereses

| Parte interesada       | Interés                                                         |
| ---------------------- | --------------------------------------------------------------- |
| Operador de seguridad  | Acceso rápido desde su terminal de trabajo                      |
| Administrador SICA     | Solo terminales registrados deben poder acceder                 |
| Auditoría              | El nombre de usuario y terminal quedan en sesión para trazabilidad |
| Seguridad              | Un terminal no registrado no debe acceder bajo ninguna circunstancia |

## Precondiciones

1. IIS tiene activada la autenticación integrada de Windows (Kerberos/NTLM).
2. La tabla `tblTerminais` contiene el hostname o IP del PC desde el que se conecta.
3. El usuario tiene cuenta de dominio activa en `refer.pt`.

## Postcondiciones (garantías de éxito)

1. `Session("Utilizador")` contiene el nombre SAM del usuario (sin dominio).
2. `Session("NomeTerminal")` contiene el nombre del terminal recuperado de `tblTerminais`.
3. La página redirige al panel de monitorización (`MonSeg.aspx`).

## Disparadores

- Carga de `Default.aspx` desde un navegador (GET o POST inicial).

## Flujo principal de éxito

| Paso | Actor     | Sistema                                                                      |
| ---- | --------- | ---------------------------------------------------------------------------- |
| 1    | Operador  | Navega a `Default.aspx` desde su terminal                                    |
| 2    |           | IIS autentica con Windows Auth → `LOGON_USER = "REFER\\usuario"`            |
| 3    |           | Extrae el SAM: subcadena tras `\` → `Session("Utilizador") = "usuario"` (RULE-009) |
| 4    |           | Timer asíncrono activa `VerificaAcesso(hostname, IP)` (RULE-008)            |
| 5    |           | Ejecuta: `SELECT nome FROM tblTerminais WHERE upper(nome)=hostname OR ip=IP` |
| 6    |           | Resultado > 0 filas: `Session("NomeTerminal") = nombre del terminal`         |
| 7    |           | Redirige a `MonSeg.aspx`                                                     |

### Detalle de pasos

**Paso 3 — Extracción del principal (RULE-009)**

- `Request.ServerVariables("LOGON_USER")` devuelve `"DOMINIO\cuenta"`.
- Si longitud > 1: `Session("Utilizador") = cadena.Substring(índice_de_\ + 1)`.
- Si longitud ≤ 1: `Session("Utilizador") = ""`.

**Paso 5 — Lista blanca de terminal (RULE-008)**

- Criterios: hostname en mayúsculas OR IP del cliente.
- ⚠️ **Inyección SQL CRÍTICA**: `NomePC` e `IP` se concatenan sin parametrizar.
  Un atacante puede falsificar su hostname para bypass total de la autorización.
  Fuente: [Default.aspx.vb:12](../../../../../demo/from_old_src/SICAWeb/SICAWeb/Default.aspx.vb).

## Extensiones (flujos alternativos)

### 5a. Terminal no registrado

| Paso | Actor    | Sistema                                                  |
| ---- | -------- | -------------------------------------------------------- |
| 5a.1 |          | Query devuelve 0 filas                                   |
| 5a.2 |          | `VerificaAcesso` devuelve cadena vacía                   |
| 5a.3 |          | La sesión no se inicializa con terminal                  |
| 5a.4 | Operador | Ve la página sin contenido o recibe error de acceso      |

### 3a. LOGON_USER vacío (autenticación fallida en IIS)

| Paso | Sistema                                                 |
| ---- | ------------------------------------------------------- |
| 3a.1 | `LOGON_USER` devuelve cadena vacía o longitud ≤ 1       |
| 3a.2 | `Session("Utilizador") = ""`                            |
| 3a.3 | El sistema no deniega acceso explícitamente (gap)       |

### 5b. Inyección SQL — Bypass de autorización (vector de ataque)

| Paso | Sistema                                                                |
| ---- | ---------------------------------------------------------------------- |
| 5b.1 | Atacante envía hostname: `'; SELECT 'terminal_admin'--`                |
| 5b.2 | Query manipulada devuelve resultado artificial                         |
| 5b.3 | `Session("NomeTerminal")` se establece → acceso total concedido ⚠️     |

## Requisitos especiales

- **Seguridad CRÍTICA**: La consulta de `tblTerminais` debe usar parámetros. Sin ello,
  un bypass en este paso concede acceso a todas las funciones del sistema.
- **Sin MFA**: Solo Windows Auth; sin segundo factor.
- **Sin RBAC por usuario**: El rol efectivo es el terminal, no el usuario.
- **Auditoría**: No se registra el evento de acceso en `tblLog`.

## Frecuencia estimada

- Cada vez que un operador abre o recarga SICAWeb (potencialmente decenas de veces al día
  por terminal).

## Cuestiones abiertas

- ¿Se valida en algún punto posterior que `Session("Utilizador")` no está vacío?
- ¿Puede un operador trabajar desde cualquier terminal registrado o hay restricciones
  operador-terminal?
- ¿Existe algún timeout de sesión configurado?
