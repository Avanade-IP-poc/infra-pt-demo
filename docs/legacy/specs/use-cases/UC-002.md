# Use Case: Clasificación y Sincronización de Tarjetas a SICAWeb

## Metadata

| Propiedad     | Valor                                                  |

| UC ID         | UC-002                                                 |
| Reglas        | RULE-004, RULE-005                                     |
| Actor primario| Servicio `SICADataSync` (Daemon)                       |
| Alcance       | Sistema — SICADataSync                                 |
| Nivel         | Subfunción (invocada tras UC-001)                      |

| Fuente        | `demo/from_old_src/SICADataSync/DataSync.vb:245-273`   |

## Descripción

Tras sincronizar usuarios, `SICADataSync` sincroniza el catálogo de tarjetas inteligentes
(smart cards) desde la tabla `Alizes.Badge` hacia la caché local `SICA.tblCartoes`. Cada
tarjeta se clasifica en uno de 5 tipos según el prefijo de su código, y solo se inserta si
no existe previamente (sin duplicación).

## Partes interesadas e intereses

| Parte interesada     | Interés                                                         |
| -------------------- | --------------------------------------------------------------- |
| Operación de Seguridad | La caché local de tarjetas refleja las emitidas en Alizes    |
| Sistema de accesos   | SICAWeb necesita conocer el tipo de tarjeta para reglas de acceso |

## Precondiciones

1. La conexión a `Alizes` (`rfsql01`) y a `SICA` está establecida.
2. La tabla `Badge` en Alizes contiene registros con `supprime = 0`.
3. La tabla `tblCartoes` en SICA existe y es accesible.

## Postcondiciones (garantías de éxito)

1. Todas las tarjetas activas (prefijo C/V/M/A/R) de Alizes existen en `tblCartoes`.
2. Cada tarjeta tiene asignado un `Tipo` numérico (1–5) según su prefijo.
3. No se crean duplicados en `tblCartoes`.

## Disparadores

- Invocado automáticamente al finalizar la sincronización de usuarios (UC-001).

## Flujo principal de éxito

| Paso | Actor  | Sistema                                                                          |
| ---- | ------ | -------------------------------------------------------------------------------- |
| 1    | Daemon |                                                                                  |
| 2    |        | Consulta `Badge JOIN Usager WHERE supprime=0 AND LEFT(codelogique,1) IN (C,V,M,A,R)` |
| 3    |        | Consulta `tblCartoes` para obtener el conjunto de `NumCartao` existentes         |
| 4    |        | Para cada badge: extrae el primer carácter del `codelogique` (RULE-004)          |
| 5    |        | Aplica la tabla de prefijos → asigna `idTipoCartao` (1–5)                        |
| 6    |        | Comprueba si `NumCartao` ya existe en `tblCartoes` (RULE-005)                    |
| 7    |        | Si no existe: INSERT (`NumCartao`, `Decricao = prenom`, `Tipo`)                  |
| 8    |        | Si existe: omite (sin actualización)                                             |
| 9    |        | Acumula contador de tarjetas insertadas                                          |

### Detalle de pasos

**Paso 4–5 — Clasificación por prefijo (RULE-004)**

| Prefijo | `idTipoCartao` | Tipo          |
| ------- | -------------- | ------------- |
| `V`     | 1              | Visitante     |
| `M`     | 2              | Mantenimiento |
| `C`     | 3              | Corporativo   |
| `R`     | 4              | Restringido   |
| `A`     | 5              | Admin         |
| (otro)  | 1              | Visitante (default) |

**Paso 7 — INSERT (RULE-005)**

- `Decricao` = campo `prenom` del usuario propietario de la tarjeta.
- ⚠️ El INSERT concatena `codelogique` y `prenom` sin parametrizar →
  inyección SQL posible ([DataSync.vb:264](../../../../../demo/from_old_src/SICADataSync/DataSync.vb)).

## Extensiones (flujos alternativos)

### 4a. Prefijo no reconocido

| Paso | Sistema                                          |
| ---- | ------------------------------------------------ |
| 4a.1 | Primer carácter no es C/V/M/A/R                  |
| 4a.2 | `idTipoCartao = 1` (Visitante) por defecto        |
| 4a.3 | El registro se procesa normalmente                |

### 2a. Badge con codelogique vacío

| Paso | Sistema                                                  |
| ---- | -------------------------------------------------------- |
| 2a.1 | `Substring(0,1)` sobre cadena vacía lanza excepción      |
| 2a.2 | Sin manejo explícito → ejecución puede interrumpirse     |

### 6a. Tarjeta ya existe en tblCartoes

| Paso | Sistema                                     |
| ---- | ------------------------------------------- |
| 6a.1 | `dtSICAWeb.Select(...)` devuelve resultado   |
| 6a.2 | Tarjeta omitida; contador no se incrementa  |
| 6a.3 | Los datos de `Decricao`/`Tipo` no se actualizan (posible desfase) |

## Requisitos especiales

- **Seguridad**: INSERT sin parametrizar — riesgo de inyección SQL (P1). ⚠️
- **Consistencia**: Cambios de tipo o descripción de tarjeta en Alizes no se propagan
  (solo se sincronizan tarjetas nuevas).
