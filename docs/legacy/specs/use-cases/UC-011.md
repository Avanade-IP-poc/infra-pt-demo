# Use Case: Edición de Detalle de Tarjeta de Visitante

## Metadata

| Propiedad        | Valor                                                       |
| ---------------- | ----------------------------------------------------------- |
| UC ID            | UC-011                                                      |
| Reglas           | RULE-006, RULE-007 (gestión de tarjetas de visitante)       |
| Actor primario   | Responsable de recepción (usuario humano en terminal)       |
| Alcance          | Sistema — SICAWeb (`DetalheVisitante.ascx`, `AtribuicaoCartoes.aspx`) |
| Nivel            | User Goal                                                   |
| Estado           | Draft — extraído de legacy                                  |
| Fuente           | `demo/from_old_src/SICAWeb/SICAWeb/DetalheVisitante.ascx.vb` |

## Descripción

El responsable de recepción busca una tarjeta de visitante, consulta y edita sus datos
(nombre, empresa, entidad visitada, tipo), gestiona horas de entrada/salida y validez, cambia
el estado de la tarjeta (Activo, Prohibida, Perdida, Robada, Destruida) y asigna familias de
acceso. Puede validar un colaborador contra Active Directory para rellenar sus datos.

## Partes interesadas e intereses

| Parte interesada       | Interés                                                         |
| ---------------------- | --------------------------------------------------------------- |
| Responsable recepción  | Registrar y mantener los datos del visitante de forma ágil      |
| Seguridad              | Estados de tarjeta correctos y familias de acceso adecuadas     |
| Visitante / Colaborador| Acceso correcto a las zonas autorizadas durante su validez      |

## Precondiciones

1. El operador ha superado [UC-003](UC-003.md) (`Session("NomeTerminal")` establecido).
2. Existe la tarjeta a editar o es una nueva asignación.
3. El servicio SMI gestiona el estado físico de la tarjeta.

## Postcondiciones (garantías de éxito)

1. Los datos del visitante quedan persistidos (`sp_tblVisitantes_Insert/Update`).
2. El estado de la tarjeta y las familias quedan sincronizados con SMI.
3. Se genera registro de auditoría asociado a `Session("Utilizador")`.

## Disparadores

- Selección de una tarjeta desde [UC-010](UC-010.md).
- Pulsación del botón de búsqueda de tarjeta.

## Flujo principal de éxito

| Paso | Actor      | Sistema                                                                |
| ---- | ---------- | ---------------------------------------------------------------------- |
| 1    | Recepción  | Busca una tarjeta por número                                          |
| 2    |            | Valida que la tarjeta tenga acceso en el terminal actual (`vwAcessos`) |
| 3    |            | Carga datos del visitante (`sp_tblVisitantes_SelectUltimaAtribuicao`)  |
| 4    |            | Sugiere validez/hora de entrada y estado Activo si es tarjeta nueva    |
| 5    | Recepción  | Edita datos, horas, estado y familias                                  |
| 6    | Recepción  | Pulsa "Gravar"                                                        |
| 7    |            | Persiste vía `sp_tblVisitantes_Insert/Update`                         |
| 8    |            | Sincroniza estado y familias con SMI                                  |

### Detalle de pasos

**Paso 2 — Validación de acceso (⚠️ riesgo de seguridad)**

- ⚠️ **Inyección SQL CRÍTICA**: `DetalheVisitante.ascx.vb:54` concatena `NumCartao` y
  `Session("NomeTerminal")` sin parametrizar sobre `vwAcessos`.

**Paso 4 — Sugerencias inteligentes**

- Tarjeta sin horas registradas → sugiere validez = hoy, entrada = ahora, estado = Activo.
- Tipos de visitante: Visitante (1), Colaborador (2), Vehículo (3).
- Estados SMI: Active→2, Forbiden→4, Lost→8, Stolen→16, Destroied→32.

**Paso 5 — Validación de colaborador en AD (⚠️ riesgo de seguridad)**

- ⚠️ **Inyección SQL CRÍTICA**: `DetalheVisitante.ascx.vb:509` concatena
  `txtVisitanteNumEmpregado.Text` sin parametrizar sobre `tblAD_AD_SQL`.
- Obtiene `displayName`, `department` y la foto (`wWWHomePage`).

## Extensiones (flujos alternativos)

### 1a. Tarjeta no encontrada

| Paso | Sistema                                          |
| ---- | ------------------------------------------------ |
| 1a.1 | La búsqueda no devuelve resultados               |
| 1a.2 | Se trata como alta de nueva asignación           |

### 5a. Colaborador con número de empleado inválido

| Paso | Sistema                                                       |
| ---- | ------------------------------------------------------------- |
| 5a.1 | `cmdValidarEmpregado_Click` no encuentra el empleado en AD    |
| 5a.2 | No rellena los campos y avisa al operador                     |

### 6a. Cancelación

| Paso | Actor      | Sistema                                          |
| ---- | ---------- | ------------------------------------------------ |
| 6a.1 | Recepción  | Pulsa "Cancelar"                                |
| 6a.2 |            | Descarta los cambios sin persistir               |

## Requisitos especiales

- **Seguridad CRÍTICA**: Parametrizar `vwAcessos` (L54) y `tblAD_AD_SQL` (L509). La
  concatenación de `NumCartao` y del número de empleado permite inyección SQL.
- **Integridad**: La sincronización con SMI debe ser consistente con la persistencia en SICA.
- **Auditoría**: Toda modificación se asocia al usuario en sesión.

## Frecuencia estimada

- Uso frecuente en recepción durante el horario de visitas.
